    • What is Ionic?

  • Ionic is a framework for creating high quality applications
  • Tradionally this has been mobile applications in a webview
  • ----arrow down------
  • It is a collection of CSS + Javascript resources
  • We call it the Ionic Framework because we provide more value than just components.
  • The code is built in such a way that its components can be used with a number of different frameworks.
  • We currently only offically support Angular and Vanilla JS
  • But this will be changing very soon
  • Ionic also provides solutions for your appliction to interact with native devices through bridges
  • We have 2 ways of accomplishing this. One of these is Cordova and the other is Capacitor
  • Capacitor is a product we built and consider to be the spiritual successor to Cordova
  • This means that you can also deploy your application to iOS, Android, and Electron. Obviously you can also ship it as a PWA

    • What is Ionic?

    • 5M + Apps created
    • 30K+ Stack Overflow Questions
    • 100+ Meetups
    • Open Source and MIT licensed
    • Collection of over 100 components
    • Cross-platform apps with a single codebase
    • Use a native bridge for device apis
  • --------------
  • We have analytics and almost every country in the world uses Ionic today.
  • Except for one. North Korea
  • --------------
  • The components range from buttons, to dialogs, to form toggles, inputs, alerts.
  • One of the things that is very unique about all of our components is that they are built for both iOS and Material Design.
  • So it is a truly cross platform codebase. Build your app once and it will appear as Material for Android and as iOS on an Apple Product.
  • We also provide a set of components and APIs that allow you to interact with native device APIs. An example of this would be push notifications, haptic feedback, geo location, camera and a number of others
    •  
  • Beautiful design - We place a lot of value on the quality of our component designs. The nice thing is that it just looks this great out of the box.
  • Notice that components reflect the native behavior for material design or for iOS such from Action Sheets or alerts.
  • Ionic also provides tabs and stacked routing which is expected for mobile applications.
  • We have 2 svg based icon sets as well. One for iOS and one for Material. You might have used them before. Aptly named Ionicons
  • Robust theming system based on css variables. So you can create a theme specific to your brand or change your themes real time in a white labeled application. This is a nice advantage over using something like sass.

    • Extensive documentation.

  • Our documentation provides live examples, best practices, and example code.
  • All supported frameworks also have code that corresponds to the live examples. - currently this is Angular and Vanilla JS
  • Ionic is written entirely in Typescript so our API documentation gets generated on release to ensure that they stay up to date. Nothing worse than outdated documenatation.

    • When should you use Ionic?

    • You are building a mobile app
    • You are building a PWA / web app
    • You are building an desktop app
  • Being built with web technologies means that Ionic is very versatle
  • --------------
  • Mobile App - WebView to access native code. Deploy to Apple App Store or Google Play
  • Web App - Shared Codebase. Not code sharing this is a single codebase. Appears native
  • Desktop App - Electron integration with Capacitor for native access.
    •  
  • Historically if you wanted to use Ionic you had to write your app in Angular.
  • Ionic was originally built in AngularJS and then Ionic 2/3 were both written in Angular with Typescript
  • Ionic has always been built in Angular.
  • But devs that moved from AngularJS to other frameworks have been asking for Ionic

    • The JS ecosystem is changing

    • "Which framework do you prefer?"
    • Ionic components should work with any framework or with no framework
  • What is the best framework, is no longer the question. The question is which framework do you prefer.
  • Whatever your setup/framework we want our components to work there. Any framework (or no framework at all)
  • We were able to accomplish this with web components.
  • Every major browser supports them now except for Edge/IE11 but we have polyfills in place for those.
